# Multi-Shell - 구조 문서

## 전체 아키텍처 개요

### 아키텍처 패턴

**단일 모듈 (Monolithic)** 방식 채택

- 초기 단계의 솔로 개발 프로젝트에 적합
- 프로젝트 성장에 따라 모듈 분리 가능한 구조로 설계
- TUI 애플리케이션 특성상 단일 바이너리 배포가 유리

### 시스템 구성도

```
+---------------------------------------------------+
|              Multi-Shell Application               |
|                                                    |
|  +-------------+  +-------------+  +-------------+ |
|  |   UI Layer  |  | Shell Mgr   |  |  Config     | |
|  |  (TUI/렌더) |  | (프로세스)   |  |  (설정관리) | |
|  +------+------+  +------+------+  +------+------+ |
|         |                |                |        |
|  +------+----------------+----------------+------+ |
|  |              Core Engine                      | |
|  |  (이벤트 루프, 입력 처리, 레이아웃 관리)       | |
|  +-----------------------------------------------+ |
|         |                                          |
|  +------+----------------------------------------+ |
|  |           Platform Abstraction Layer           | |
|  |  (OS별 셸 프로세스, PTY, 터미널 제어)          | |
|  +-----------------------------------------------+ |
+---------------------------------------------------+
```

## 주요 모듈 및 책임

### 1. Core Engine (`src/app.rs`)

- 애플리케이션 메인 이벤트 루프 관리
- UI 업데이트 주기 제어 (목표: 60fps)
- 모듈 간 이벤트 라우팅
- 글로벌 상태 관리

### 2. UI Layer (`src/ui/`)

- TUI 프레임워크를 사용한 터미널 렌더링
- 패널 분할 레이아웃 계산 및 적용
- 각 셸 출력의 가상 터미널 렌더링
- 상태바, 포커스 표시, 셸 제목 표시

### 3. Shell Manager (`src/shell/`)

- 셸 프로세스 생성, 모니터링, 종료
- PTY(의사 터미널) 할당 및 관리
- 셸 I/O 스트림 비동기 처리
- 플랫폼별 셸 추상화 (Windows: ConPTY, Unix: POSIX PTY)

### 4. Input Handler (`src/input/`)

- 키보드 입력 캡처 및 라우팅
- 모드 기반 입력 처리 (일반 모드 vs 명령 모드)
- 단축키 바인딩 관리
- 마우스 이벤트 처리 (선택적)

### 5. Config (`src/config/`)

- TOML 형식 설정 파일 파싱
- 기본값 제공 및 사용자 오버라이드
- 런타임 설정 변경 지원

## 데이터 흐름

### 입력 처리 흐름

```
사용자 키 입력
    → Input Handler (단축키 vs 일반 입력 분류)
    → [단축키] → Core Engine → UI/Shell 명령 실행
    → [일반 입력] → 활성 셸 패널의 PTY로 전달
```

### 셸 출력 흐름

```
셸 프로세스 출력 (PTY stdout)
    → Shell Manager (비동기 수신)
    → 출력 버퍼 저장 (스크롤백 히스토리)
    → Core Engine (UI 업데이트 트리거)
    → UI Layer (해당 패널에 렌더링)
```

## 의존성 방향

```
main.rs → app.rs (Core Engine)
    ├── ui/ (UI Layer)
    ├── shell/ (Shell Manager)
    ├── input/ (Input Handler)
    └── config/ (Configuration)
```

- 모든 모듈은 Core Engine을 통해 간접적으로 통신
- UI와 Shell 사이에 직접적 의존성 없음
- Config는 다른 모든 모듈에서 읽기 전용으로 참조

## 아키텍처 결정 배경

| 결정 | 근거 |
|------|------|
| Rust 선택 | 크로스 플랫폼 바이너리, 메모리 안전성, 높은 성능 |
| 단일 모듈 구조 | 솔로 개발, 초기 단계 복잡도 최소화 |
| PTY 기반 셸 관리 | 실제 터미널 에뮬레이션으로 셸 호환성 보장 |
| 이벤트 기반 아키텍처 | TUI 애플리케이션의 표준 패턴, 비동기 I/O 지원 |
| TOML 설정 | Rust 생태계 표준, 가독성 우수 |

## 비기능 요구사항

| 항목 | 목표 | 비고 |
|------|------|------|
| 성능 | UI 렌더링 16ms 이하 (60fps) | TUI 특성상 CPU 부하 최소 |
| 메모리 | 셸당 50MB 이하 | 스크롤백 버퍼 크기 제한 |
| 시작 시간 | 1초 이내 | Rust 네이티브 바이너리 |
| 플랫폼 | Windows 11 (우선), macOS, Linux | ConPTY/POSIX PTY 추상화 |
| 접근성 | 키보드 전용 조작 가능 | 마우스 지원은 선택적 |
